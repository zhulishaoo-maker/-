# AI Native 电商会场自动化实施计划

## 1. 实施目标

以 [AI Native 电商会场自动化产品设计](../specs/2026-07-13-ai-native-venue-operations-design.md) 为唯一需求基线，交付一个可试点的 MVP：运营输入活动 Brief，AI 生成受控会场，经审核和灰度后上线，再通过实验与护栏完成动态调权。

## 2. 建议技术基线

当前仓库尚无代码。为便于快速验证和后续拆分，MVP 建议采用 TypeScript 模块化单体 + 异步事件工作进程：

- Web：Next.js + React，同时承载运营工作台和会场渲染器。
- API：NestJS 或 Fastify；如团队已有 Node.js 框架规范，优先沿用。
- 数据库：PostgreSQL，使用 JSONB 存储版本化 Page Schema。
- 队列与缓存：Redis + BullMQ，用于 AI 任务、发布、指标汇总和回滚。
- 对象存储：S3 兼容存储，用于素材和发布快照。
- 事件采集：MVP 先使用 HTTP 事件入口 + 队列；数量级上升后再替换为 Kafka/ClickHouse。
- AI 接入：模型无关的 Provider 接口，所有工具调用通过服务端白名单和 Schema 校验。
- 观测：OpenTelemetry + 结构化日志 + 错误告警。

在开发前与团队确认现有账号、商品、权益、素材、发布和数据平台的接入协议。若已有标准技术栈，保留本计划的模块边界，替换具体框架即可。

## 3. 目标仓库结构

```text

  ops-web/                 # 运营工作台
  venue-web/               # 会场渲染与事件上报
  api/                     # 业务 API 与权限
  worker/                  # AI、发布、实验、指标任务

  page-schema/             # Schema、迁移、校验器
  component-registry/      # 组件元数据与默认参数
  renderer/                # Schema -> React 渲染
  domain/                  # 领域模型与用例
  agent-runtime/           # 规划、工具、记忆、追踪
  rules-engine/            # 发布规则和护栏

  fixtures/
  contracts/
  e2e/
infra/
  migrations/
  local/

```

## 4. 核心合同

### 4.1 Page Schema

Page Schema 必须包含：

- `schemaVersion`、`campaignId`、`pageVersionId`。
- 页面元数据：标题、主题、生效时间、目标人群。
- 组件列表：`id`、`type`、`props`、`contentRefs`、`locked`、`trackingId`。
- 规则和实验引用：只存标识，不将敏感逻辑下发至客户端。
- 生成来源：人工、AI 候选或实验变更。

任何 Schema 发布前必须通过 JSON Schema 校验、组件合同校验和业务规则校验。

### 4.2 智能体工具协议

每个工具必须定义：

- 唯一名称和版本。
- 严格输入/输出 Schema。
- 所需权限和运行时审批等级。
- 超时、重试、幂等键和错误分类。
- 审计记录和敏感数据脱敏规则。

初始工具集：`search_products`、`search_assets`、`get_benefits`、`get_campaign_history`、`create_page_candidate`、`validate_page`、`render_preview`、`submit_for_review`、`query_metrics`、`propose_experiment`。AI 不拥有直接全量发布或修改交易配置的工具。

## 5. 实施阶段

### 阶段 0：工程基座和接入合同

**目标：** 建立可持续开发、测试和演示的工程骨架。

1. 初始化 monorepo、TypeScript、lint、格式化、单测和 CI。
2. 创建 API、工作台、会场渲染器和 worker 空应用。
3. 建立本地 PostgreSQL、Redis 和对象存储环境。
4. 定义商品、权益、素材和身份服务的适配器接口及 mock provider。
5. 接入 OpenTelemetry、结构化日志和全链路 `requestId`。

**验收：** 一条命令启动本地环境；CI 执行 lint、类型检查和测试；四个应用健康检查通过。

### 阶段 1：Schema、组件库和渲染器

**目标：** 在不接入 AI 的情况下，从受控 Schema 稳定渲染会场。

1. 定义 Page Schema v1、迁移接口、校验器和 fixture。
2. 建立组件注册表，每个组件声明 props Schema、默认值、可调参数和风险等级。
3. 实现首批 8–12 类会场组件。
4. 实现 Schema 到 React 的渲染器，未知组件和非法参数安全降级。
5. 为每个组件注入统一的曝光和交互埋点。
6. 建立组件视觉回归和移动端性能基线。

**验收：** 固定 fixture 能产生稳定页面；非法 Schema 无法进入发布链路；埋点包含 Campaign、PageVersion、Component 和 Experiment 标识。

### 阶段 2：手工会场全链路

**目标：** 先证明确定性执行面安全可用。

1. 创建 `Campaign`、`Brief`、`PageVersion`、`Component`和审批数据库表及迁移。
2. 实现活动列表、Brief 表单、组件编辑和实时预览。
3. 实现页面差异比较、组件锁定和版本创建。
4. 实现 RBAC：运营、业务审批人、管理员。
5. 实现发布前规则链：权益、价格、库存、素材授权、品牌。
6. 实现灰度发布、不可变快照、健康校验、全量和一键回滚。
7. 实现静态安全页和回滚失败告警。

**验收：** 不使用 AI 即可完成创建、审核、灰度、上线和回滚；完整 E2E 通过；所有发布动作可审计。

### 阶段 3：会场运营智能体 L0/L1

**目标：** 实现 AI 补全 Brief、生成候选和提出建议，但不自动发布。

1. 实现 Provider 接口、模型配置、超时、重试和降级策略。
2. 实现工具注册、Schema 校验、权限检查、幂等和审计。
3. 实现 Brief 解析：可查得数据自动补全，仅对阻断项追问。
4. 实现稳健、均衡、探索三类候选生成。
5. 将 AI 输出限制为 Page Schema patch，通过规则链后才可预览。
6. 实现候选对比、决策理由、证据引用、预期和风险界面。
7. 实现自然语言修改，确保锁定组件不受影响。
8. 建立离线评估集，持续评估结构合法性、工具选择、规则遵循和可解释性。

**验收：** 候选结构/确定性业务规则通过率达 95%；AI 无全量发布权限；所有决策可追溯；超时和无效输出不影响当前稳定版。

### 阶段 4：实时数据和实验引擎

**目标：** 建立可信的反馈和归因基础。

1. 定义行为、转化、经营和系统事件协议。
2. 实现事件接收、去重、质量校验、聚合和数据新鲜度标记。
3. 创建 `MetricSnapshot`、`Experiment` 和实验分组数据模型。
4. 实现稳定分流、同期对照、流量上限和实验互斥。
5. 实现最小样本量、置信度、最小业务效应量和多指标护栏判断。
6. 实现活动指标、实验状态、分组对比和数据质量看板。
7. 建立埋点丢失、延迟和流量分布异常告警。

**验收：** 固定用户稳定进入同一分组；数据不完整时禁止判胜；历史流量回放能复现预期实验结果。

### 阶段 5：护栏、L2/L3 和自动回滚

**目标：** 将 AI 从建议模式逐步升级为护栏内自治。

1. 实现可版本化的护栏规则和 `GuardrailEvent`。
2. 实现自治等级、活动级授权、手工升级和自动降级。
3. 实现 AI 实验提案：假设、证据、变量、人群、流量、指标和停止条件。
4. 限制每次实验只改一类变量，并校验核心权益和锁定组件未变。
5. 实现小流量自动启动、逐级扩量、停止和回滚状态机。
6. 实现护栏越界自动停流与回滚，并完成故障注入演练。
7. 将经验证结论写入带人群、时间和环境限定的策略记忆。

**验收：** 自动变更 100% 可审计；护栏触发后 1 分钟内停止新实验流量；回滚失败时可切换安全页。

### 阶段 6：试点、评估与上线

**目标：** 使用可控流量验证产品价值和运行安全。

1. 选择 2–3 个风险较低、流量足够的电商活动。
2. 使用历史 Brief 和人工会场执行离线回放，设置人工基线。
3. 第一轮仅开启 L0/L1，验证生成时间、规则通过率和运营接受度。
4. 第二轮开启 L2，对一类低风险调权执行预注册实验。
5. 在连续稳定运行、回滚演练和安全评审通过后，仅对已验证策略开启 L3。
6. 输出试点报告：效率、质量、经营效果、护栏事件、人工干预和后续范围。

**验收：** 达到规格文档第 13 节定义的 MVP 成功标准，否则保持 L0/L1 并进入改进周期。

## 6. 关键测试矩阵

| 对象 | 单元/属性测试 | 合同/集成测试 | E2E/故障测试 |
|---|---|---|---|
| Page Schema | 非法 props、迁移幂等 | 注册表与渲染器兼容 | 未知组件安全降级 |
| 规则引擎 | 规则优先级、阈值边界 | 商品/权益/素材适配器 | 发布阻断与告警 |
| 审核发布 | 状态迁移、权限 | 快照、灰度、CDN | 并发发布、回滚失败 |
| 智能体 | 工具选择、输出校验 | Provider 超时与重试 | 无效输出不影响线上版 |
| 事件数据 | 去重、窗口聚合 | 采集到指标链路 | 延迟、丢数据、乱序 |
| 实验 | 稳定分流、样本门槛 | 对照保留和归因 | 流量异常时暂停判胜 |
| 护栏 | 阈值、去抖和恢复 | 停流、回滚、降级 | 故障注入和安全页 |

## 7. 交付关卡

每个阶段进入下一阶段前，必须同时满足：

- 代码审查和自动化测试通过。
- 新增合同和运维操作已文档化。
- 权限、审计和敏感数据评审通过。
- 可对新功能执行安全降级或回滚。
- 指标与告警已可用，且有明确责任人。

## 8. 建议团队配置

- 1 名产品负责人/电商运营专家。
- 1 名技术负责人。
- 2 名前端工程师（工作台 + 渲染器）。
- 2 名后端工程师（领域、发布、集成）。
- 1 名 AI 工程师（智能体、工具、评估）。
- 1 名数据工程师/实验平台工程师。
- 1 名测试工程师，并由 SRE/安全人员阶段性参与。

## 9. 主要风险与应对

- **现有系统接入不稳定：** 先定义适配器合同和 mock，将外部变化隔离在 `integrations`。
- **AI 候选可用但不好用：** 建立运营接受率、修改量和决策理由质量评估，不只看 Schema 通过率。
- **数据偏差导致错误调权：** 对照组、最小样本、数据质量标记和人工升级同时存在。
- **过度追求综合得分：** 保留原始指标，任一硬护栏可独立否决。
- **自治权限扩张过快：** 活动级授权，手工升级，系统只能自动降级。
- **无法证明 AI 带来增量：** 试点前预注册指标、最小效应量和分析方法。

## 10. 开工前必须确认的外部依赖

以下项目不影响开始阶段 0/1，但必须在对应集成开发前确认：

- 身份系统、角色和审批人来源。
- 商品、价格、库存、权益和素材 API 及 SLA。
- 现有发布、CDN、灰度和回滚基础设施。
- 用户标识、人群标签和数据合规限制。
- 支付、毛利和补贴成本指标的归因口径。
- 首批试点业务、人工基线和风险责任人。

## 11. 第一个开发迭代的建议任务

1. 初始化 monorepo 和 CI。
2. 创建 Page Schema v1 及测试 fixture。
3. 创建组件注册表和 3 个样板组件：权益头图、商品网格、优惠券。
4. 创建 Schema 渲染器和统一埋点接口。
5. 完成一个固定 fixture 会场的移动端渲染、视觉回归和性能基线。

该迭代结束后，团队将拥有整个产品的最小稳定核心：一个可版本化、可校验、可渲染的会场 Schema。
